Methods

This page documents how data is structured, displayed, and analyzed in the Alaska Landslide Inventory viewer. It covers the classification system, how map features are positioned, how areas and volumes are derived, the automated rules that maintain consistency across columns, and how the two timeline charts work.

Rule names throughout this page link to a detail page showing the actual function source (Python or SQL) and a live diff between the rule's output and the current database. Editors can additionally apply rule changes from there; the page is otherwise read-only.

1. Data sources and structure

The inventory lives in a PostGIS database (tethys_db) with two primary tables: landslides (one row per landslide, all attributes) and landslide_polygons (one row per polygon, with a landslide_id foreign key). Every landslide has at least one polygon; catastrophic landslides typically have two or more (separate source and deposit geometries).

  • Slow landslides (~1,110 records) — one body polygon each, representing the full mapped extent of the moving mass.
  • Catastrophic landslides (~314 records, ~620 polygons) — polygons tagged as source (headscarp / failure initiation zone) or deposit (runout / debris field). A typical catastrophic landslide has one source and one deposit; some have only a deposit (the source may be obscured or uncertain), and a few large events have multiple polygons of the same role.

Source geometry is stored in WGS 84 (EPSG:4326) for compatibility with web standards. Area, volume, and centroid computations are performed in Alaska Albers Equal Area (EPSG:3338) via PostGIS ST_Transform — an equal-area projection avoids the 4–6× area distortion that Web Mercator introduces at Alaskan latitudes.

The complete inventory can be downloaded as GeoJSON + QGIS .qml style files from the ⬇ Download link in the header. The download includes a manifest, the normalized landslides and polygons GeoJSONs, a denormalized flat polygons file with parent landslide attributes merged in, and style files matching this site's map symbology.

Each landslide carries a unique, human-readable unique_name. Because Alaska placenames recur across distinct features, names follow a compact disambiguation standard — see Naming conventions.

2. Classification system

Every landslide has a type (slow or catastrophic) and a class that encodes both the type and the strength of evidence for ongoing creep (for active landslides) or its age (for prehistoric events). Classes are organized in the sidebar by activity and recency.

Catastrophic landslides with precursory creep are drawn with a colored halo (stroke) matching their creep evidence level. The “Catastrophic Cryptic” class (no halo) marks post-2012 events whose pre-event data were reviewed and no precursory creep was found.

The classification of any individual landslide is itself derived by an automated rule from a few input columns — see Automated rules below.

3. Polygons and map placement

Polygon roles

Each polygon in the database has a role:

  • body — the single polygon for a slow landslide, representing its full mapped extent.
  • source — the headscarp or failure initiation zone of a catastrophic event.
  • deposit — the runout or debris deposit of a catastrophic event.

The is_primary flag

Every landslide that has been digitized to completion carries exactly one primary polygon, used for dot placement on the map:

  • Slow: the body polygon is flagged is_primary=true. All slow landslides in the inventory have a primary body polygon.
  • Catastrophic with a source: the source polygon carries is_primary=true. Multi-source events have one source flagged primary, the others unflagged. Deposit polygons are never flagged primary.
  • Catastrophic without a source: a small number of catastrophic records have no source polygon (the source may be obscured or hasn't been digitized). These records have no primary polygon at all; the centroid rule falls back to the deposit polygon for dot placement.

Centroid placement

Each landslide's dot is drawn at the centroid of a single representative polygon, computed in EPSG:3338 then reprojected to WGS 84:

  1. For slow landslides: centroid of the body polygon.
  2. For catastrophic landslides: centroid of the source polygon (which carries is_primary=true). If no source polygon exists, the centroid is computed from the deposit polygon as a fallback.

The centroid coordinates are stored as four derived columns on the landslides table: centroid_albers_x / centroid_albers_y (meters in EPSG:3338) and centroid_lat / centroid_lon (decimal degrees in WGS 84). They are populated by automated rules (centroid_albers_x centroid_albers_y centroid_lat centroid_lon) and re-applied whenever polygon geometries change.

Polygons themselves are drawn when zoomed in past zoom level 7.

4. Areas and volumes

Polygon areas

Per-polygon areas (stored in landslide_polygons.area) are computed by PostGIS as ST_Area(ST_Transform(geom, 3338)) and rounded to whole square meters. The Alaska Albers projection guarantees equal-area accuracy across the inventory's spatial extent. polygon_area rule

Landslide-level role aggregates

Polygon areas are summed by role into three columns on the landslides table:

  • area_body — sum of body-polygon areas. NULL for catastrophic landslides. area_body rule
  • area_source — sum of source-polygon areas. NULL for slow landslides. Handles multi-source catastrophic landslides naturally. area_source rule
  • area_deposit — sum of deposit-polygon areas. NULL for slow landslides. area_deposit rule

Size threshold for inclusion

The boolean size_inclusion column marks whether a landslide meets the inventory's size criterion:

  • Slow: area_body ≥ 50,000 m²
  • Catastrophic: area_source ≥ 50,000 m² OR area_deposit ≥ 500,000 m²

Landslides below the threshold are tagged "Small slow landslide" or "Small catastrophic landslide" by the landslide_class rule, and the size_inclusion flag is itself populated by the size_inclusion rule.

Volume estimation

Per-polygon volume estimates (landslide_polygons.polygon_volume) branch on the parent landslide's type and the polygon's role:

ConditionFormulaRationale
creeping_permafrost_mass = TRUE 20 × area Uniform 20 m thickness assumed for permafrost masses, regardless of role
Slow, role = body 0.1 × area1.5 Empirical scaling of landslide volume with planform area
Catastrophic, role = source 0.1 × area1.5 Same scaling applied to the source area
Catastrophic, role = deposit 2 × area Minimum 2 m deposit thickness assumed

polygon_volume rule

Per-polygon volumes are summed by role into volume_body volume_body, volume_source volume_source, and volume_deposit volume_deposit on the landslides table.

Estimated vs preferred volume

volume_estimated is the landslide-level rule output:

  • Slow: volume_body
  • Catastrophic: max(volume_source, volume_deposit) — the two role estimates can disagree because of deposit run-out, entrainment, or removed material; the larger is the more defensible estimate of failure size.

volume_estimated rule

Some landslides have a manually-determined volume_site_specific (from a field study, DEM differencing, or published paper). When present, it supersedes the formula-based estimate. volume_preferred is the resulting canonical value used on the map and in downstream displays:

  • If volume_site_specific is set: use it directly
  • Otherwise: use volume_estimated

volume_preferred rule

The volume_method column carries either a one-off description of a site-specific approach (paper citation, DEM-differencing note, etc.) or the canonical string “Automated estimate” for rows using the rule-driven value. volume_method rule The “Site-specific volume” filter checkbox restricts the map to landslides with a manually-entered volume_site_specific.

Slider scales

Area and volume sliders use a logarithmic scale:

  • Area: slider range 0–6 maps to 103–109 m² (1,000 m² to 1,000 km²).
  • Volume: slider range 0–8 maps to 104–1012 m³ (10,000 m³ to 1 km³).

Landslides with no area or volume data always pass these filters — the filter only excludes records where the value is known and below the threshold.

5. Automated rules and audit

Many columns are derived from other columns by deterministic rules. Each rule is a pure function (Python or SQL) implemented in inventory/derived.py and exposed at /inventory/manage/rules/ for editors. The displayed rule source is read at request time via inspect.getsource() — the displayed version and the executed version cannot drift apart.

For each rule, the audit page shows the current count of agreements and disagreements between stored values and computed values, with a row-by-row diff including a Δ % column for numeric rules. Editors can apply rule output to the whole inventory in one transaction; the bulk apply writes an audit log entry per affected landslide and invalidates the appropriate caches.

Rule dependency chains

Rules are organized into dependency chains. Within a chain, earlier rules must be applied before later rules read meaningful values.

ChainSteps
Areas polygon_area rule → area_body rule / area_source rule / area_deposit rule → size_inclusion rule
Centroids Independent — centroid_albers_x, centroid_albers_y, centroid_lat, centroid_lon (x y lat lon) (all read polygon geometry directly)
Volumes polygon_volume rule → volume_body rule / volume_source rule / volume_deposit rule → volume_estimated rule → volume_preferred rule → volume_method rule
Creep evidence → class insar_creep rule → creep_behavior rule → landslide_class rule

Notable rule logic

insar_creep rule is a tri-state OR-merge of the per-study booleans (insar_schaefer, insar_kim, insar_opera, insar_other): True if any positive reading; False if at least one study returned negative and none positive; NULL if no study has assessed the site. This preserves the distinction between "we checked, no creep" and "we haven't checked yet".

creep_behavior rule picks the label observed for this landslide from the per-source flags. The two Planet flags work as a pair, not a simple severity ladder: planet_labs_creep says “creep is apparent in Planet imagery,” and planet_labs_patchy_creep is a refinement saying “the creep is patchy, not uniform across the feature.” Conceptually obvious creep is the broader assertion (the whole feature is moving), but the patchy flag, when set, overrides that to mean “actually it’s only patchy” — so in the compositional rule the patchy flag wins:

  1. planet_labs_patchy_creep → Patchy obvious creep
  2. planet_labs_creep → Obvious creep
  3. insar_creep → Subtle creep
  4. other_subtle_creep (any non-empty value) → Subtle creep
  5. geomorph_creep → Geomorph creep
  6. none of the above, but creep_evaluated is TRUE → Cryptic — a deliberate null finding (the record was reviewed and no precursor was identified)
  7. otherwise → None (no review yet)

The creep_evaluated boolean is human-set in the data-entry UI and is not itself a rule output. It distinguishes “reviewed and nothing found” (TRUE) from “not reviewed yet” (FALSE), which matters in particular for older catastrophic events where there generally was no pre-event data to review at all.

landslide_class rule composes the class from landslide_type, size_inclusion, creep_behavior, and event year:

  • Slow + not in size scope → "Small slow landslide". Slow + in size scope → "Slow {creep_behavior}".
  • Catastrophic + not in size scope → "Small catastrophic landslide". Catastrophic + in size scope branches on event year:
    • 2012 or later: "Catastrophic {creep_behavior}" — including the "Catastrophic Cryptic" case when the creep_behavior rule has marked the record as cryptic (pre-event data reviewed, no precursor found).
    • 1850 to 2011: "Catastrophic Modern".
    • Before 1850, or year_text contains "Holocene": "Catastrophic Holocene".

Records that the rule can't classify (e.g., a slow landslide with no positive creep evidence in any source) are flagged in the legend under Other → Incomplete classification. Logged-in editors can click the ⚙ gear in the map detail popup to open the record for editing.

6. Temporal data and year encoding

Temporal precision varies enormously across the inventory. The system encodes each landslide's age as a year number using this priority:

  1. landslide_class contains "Holocene" → year = −1 (prehistoric)
  2. landslide_class contains "Modern" → year = 0 (post-settlement but pre-2000)
  3. year_text is a four-digit year → that year (e.g., 2013)
  4. date_min exists → year extracted from it

The Min age slider maps positions to this encoding: position 0 = all, 1 = Holocene+, 2 = Modern+, 3–16 = 2012–2025. This provides a coarse temporal filter that works despite the mix of precise dates and order-of-magnitude age estimates.

Precise dates

Catastrophic landslides may have:

  • Seismic datetime — precise to the second, from seismic detection (stored as UTC).
  • Date range (date_min / date_max) — a bracket: the event occurred sometime between these dates. The span can be hours (for well-observed events) to months or years.

7. Seasonal timing chart

The seasonal histogram shows when in the year catastrophic events occur. Only landslides with seismic datetimes or valid date brackets are included.

Binning

The year is divided into 37 bins: 36 bins of 10 days each (days 1–10, 11–20, …, 351–360) plus one shorter bin for the remaining 5–6 days. The final bin's bar height is normalized to a 10-day equivalent so it is visually comparable.

Weight distribution

  • Point events (seismic datetime): full weight of 1.0 assigned to the bin containing that day of year.
  • Range events (date bracket): weight is spread proportionally across all bins the range overlaps. Each bin receives weight = (days of overlap with bin) / (total span in days). A 30-day bracket contributes 1.0 total, distributed across the ~3 bins it touches, rather than counting once in each bin.

Max uncertainty slider

The "Max uncertainty" slider (default: 30 days) excludes range events with spans longer than the slider value. Setting it to 0 shows only seismically detected events.

Volume weighting and log scale

Two checkboxes change the y-axis:

  • volume weighted: each event contributes its volume_preferred rather than a count of 1. The largest single event in each bin is highlighted as a darker stripe at the top of that bar.
  • log: switches the y-axis to base-10 log, useful when one or two large events dominate.

8. Timeline chart

The timeline histogram shows when in history landslides occurred. All landslides with resolvable temporal data are included (not just time-bracketed events).

Bin structure

The chart uses variable-width bins reflecting increasing data completeness in recent periods:

PeriodBin widthVisual width
Holocene (year = −1)~10,000 years27 units
Modern pre-2000 (year = 0)~150 years45 units
Annual 2000–20111 year each9 units each
Monthly 2012–present1 month each1 unit each

The 2012+ zone is highlighted with a blue background, indicating the period of most complete satellite coverage and systematic inventory.

Rate normalization

Raw counts are converted to events per year:

  • Holocene count ÷ 10,000
  • Modern count ÷ 150
  • Annual bins (2000–2011): already per-year
  • Monthly bins × 12 to annualize

The Holocene and Modern divisors are approximate — order-of-magnitude normalizations for visual comparison, not precise rate estimates.

Monthly spreading (2012+)

  • Seismic point: assigned entirely to the month of the seismic detection.
  • Date range: weight is spread proportionally across months based on the fraction of the range falling in each month.
  • Year only: spread equally across all 12 months of that year (1/12 each).

9. Filter mechanics

All filtering is performed client-side on the full dataset. The map and both charts respond to the same set of filters.

Type and class checkboxes

Type (Slow / Catastrophic) and class checkboxes are combined with AND logic. The class checkbox list always renders every known class; classes with zero matching records appear dimmed and disabled so future data gaps are visible rather than silently hidden. Records with NULL or empty landslide_class are caught by the Other → Incomplete classification checkbox, which would otherwise filter them out entirely.

Include-only checkboxes

When checked, these restrict the display to landslides having a specific attribute. Multiple checked boxes are AND'd. Current options:

  • Molards — molards = TRUE
  • Dammed streams — stream_damming is a non-empty string
  • Precursory headscarp — precursory_headscarp = TRUE
  • Site-specific volume — volume_site_specific is set
  • Exclusively supraglacial — exclusively_supraglacial = TRUE
  • Creeping permafrost mass — creeping_permafrost_mass = TRUE
  • Time bracketed — both date_min and date_max are set
  • Seismic detection — seismic_datetime is set
  • Post-2012 activity increase — post_2012_activity_increase = TRUE

Area, volume, and age sliders

These apply a minimum threshold using ≥ comparison. Landslides with missing data for the filtered attribute pass through unconditionally.

Viewport limiting

Both charts always restrict to the current map viewport. The sidebar counts can optionally be restricted to the viewport via the "Limit counts to map view" checkbox.

URL state

All filter settings, slider positions, and panel visibility are encoded in the URL query string — filtered views can be shared as links that reproduce the exact same view.

10. Known limitations

  • Completeness varies by era. The inventory is most complete for the post-2012 satellite era. Holocene and Modern event counts are severe underestimates of actual occurrence rates. The timeline normalization makes this visually apparent but the magnitudes should not be taken as true rates.
  • Volume estimates use a simple empirical formula. The rule-driven volume_estimated uses 0.1 × area1.5 for slow body and catastrophic source polygons. This is an order-of-magnitude approximation; real volumes can deviate substantially. When a site has been independently studied, volume_site_specific supersedes the formula, and those records are filterable via the "Site-specific volume" checkbox.
  • Permafrost thickness is assumed. Landslides flagged creeping_permafrost_mass get a uniform 20 m thickness in the volume formula. Real permafrost mass thicknesses are not measured here.
  • Catastrophic deposit thickness is a floor. The 2 m deposit-thickness assumption under-estimates volume for thick deposits. For events of interest, prefer the site-specific value when present.
  • Temporal precision varies widely. Some events are timed to the second (seismic); others have year or century-scale uncertainty. The seasonal histogram's "max uncertainty" slider helps manage this.
  • Holocene and Modern rate divisors. 10,000 years for Holocene and 150 years for Modern are reasonable approximations but not tied to precise epoch boundaries.
